Cyberesa ⇄ BoosterBC — Bridge d'intégration

Un pont léger qui expose l'API Cyberesa HotelSearch (XML/SOAP-like sur strRQ), traduit la requête vers le format REST JSON de BoosterBC, appelle l'endpoint /hotels/availability, puis retraduit la réponse en <HotelSearchRES> Cyberesa.

Sortie

Auth upstream
apiKey + SHA-512

1. Vue d'ensemble

Le bridge est une application ASP.NET Core minimal API. Il accepte les mêmes routes et le même payload que la plateforme Cyberesa (POST /cr.ws.clicngo.uapi/hotel/webservicerq avec strRQ=<xml>), ce qui permet à tout client Cyberesa existant de cibler le bridge sans modification.

En interne, une couche de traduction convertit l'enveloppe XML HotelSearchREQ en paramètres query BoosterBC, signe la requête (SHA-512 de apiKey + password + timestamp) puis mappe la réponse JSON availabilityRS vers HotelSearchRES. Un mock interchangeable permet de tester sans clé valide.

6. Authentification BoosterBC

Chaque appel ajoute trois headers. La signature expire vite (horodatage contrôlé serveur) ; elle est régénérée à chaque requête. 

// trois headers requis
apiKey:    "<apiKey>"
timestamp: "<unix seconds UTC>"
signature: SHA-512( apiKey + password + timestamp )  // hex lower-case

Config dans appsettings.json (section BoosterBC). Pour une exécution locale sans clé valide, activer "UseMock": true : le bridge injecte MockBoosterClient via DI et renvoie deux hôtels de test.

7. Journalisation des échanges upstream

Chaque appel Booster (réel ou mock) produit deux fichiers sous bin/Debug/net10.0/logs/booster/ : 

20260419_101234_567_ab12cd_REQ.txt    # URL + headers (apiKey, timestamp, signature)
20260419_101234_567_ab12cd_RES.json   # HTTP status + body JSON indenté

Le préfixe <UTC>_<id6hex> est partagé par la paire REQ/RES. Les mêmes blocs apparaissent aussi en stdout (préfixés ===== BOOSTER REQ/RES =====).

9. Points d'extension

Ajouter une opération

Pour couvrir HotelQuote / HotelBookingCreation etc. : ajouter une route dans Program.cs, un POCO XML dans Models/, une méthode dans IBoosterClient (/hotels/checkRate, /hotels/book) et un mapping dans Translator.

Remplacer le mock

Le mock est un simple IBoosterClient. On peut le remplacer par un stub WireMock / WireMock.Net pour rejouer des fixtures, ou par un enregistreur de cassettes (VCR) à partir des fichiers logs/booster/.

Observabilité

Ajouter AddOpenTelemetry() pour exporter les traces (l'appel entrant, la traduction, l'appel sortant). Le logger fichier actuel reste utile pour rejeu offline.

Sécurité

Valider les creds <Credential> côté bridge (allowlist de logins, IP whitelist). Aujourd'hui, le bridge ne ré-authentifie pas le client Cyberesa : il faut l'ajouter avant mise en prod.
